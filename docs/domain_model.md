# Domain Model

Agentic Co-Emergence v0.2

## Purpose

This document defines the core conceptual objects used by Agentic Co-Emergence v0.2.

It is not a class design, implementation plan, or API specification. Its purpose is to establish the vocabulary of the project before code is written.

The runtime exists to transform a fixed set of independent perspectives into an evolving collective understanding through governed interaction.

---

## Core Principle

The central object of this project is not the transcript.

The central object is **understanding**.

A discussion is the mechanism by which understanding changes.

```text
Initial Perspectives
        │
        ▼
Governed Discussion
        │
        ▼
Evolved Understanding
```

## Question

A **Question** is the issue, problem, or proposition to be explored.

It establishes the shared object of attention.

A question should be treated as immutable during a discussion run. Participants may reframe, challenge, clarify, or decompose it, but the originating question remains part of the historical record.

### Responsibilities

A Question:

- defines the inquiry boundary
- gives all perspectives a common reference point
- anchors the discussion trace
- provides the basis for evaluating relevance

### Example

```text
Who do you think is, or was, the most underrated Western philosopher, and why?
```

---

## Perspective

A **Perspective** is an agent's independent initial understanding of the Question before interaction begins.

A Perspective is not a discussion contribution. It is a pre-discussion position.

It should be treated as immutable once captured.

### Responsibilities

A Perspective records:

- the agent's initial answer or position
- the reasons supporting that position
- relevant assumptions
- implicit criteria used by the agent
- the agent's starting interpretive frame

### Important Distinction

```text
Perspective = what the agent initially thinks
Contribution = what the agent says during discussion
Agent State = how the agent's understanding evolves
```

---

## Perspective Pack

A **Perspective Pack** is a collection of independent Perspectives addressing the same Question.

It is the canonical handoff object between Perspective Capture and Agentic Co-Emergence.

The Co-Emergence runtime does not need to know how the Perspective Pack was produced. It may have been generated by agents, written by humans, extracted from documents, or assembled from historical sources.

### Responsibilities

A Perspective Pack:

- stores the originating Question
- stores metadata such as creation time
- stores one or more named Perspectives
- provides the initial conditions for discussion
- establishes the pre-interaction diversity of thought

### Immutability

The Perspective Pack should not be modified by the discussion runtime.

The runtime may derive Agent States from it, but it should preserve the original pack as evidence of the starting condition.

---

## Agent

An **Agent** is a participant capable of making contributions to a discussion.

In v0.2, an agent may be implemented using an LLM, but the conceptual model should not depend on that. A future system could include human participants, rule-based participants, expert systems, or archived textual viewpoints.

### Responsibilities

An Agent:

- begins with an initial Perspective
- participates through Contributions
- may revise its current understanding
- may challenge, support, clarify, or synthesize
- operates within the runtime's governance rules

---

## Agent State

**Agent State** is the mutable discussion-time state of an Agent.

It is derived from the initial Perspective but changes as the discussion proceeds.

### Responsibilities

Agent State may include:

- current position
- confidence level
- commitments
- questions for other agents
- objections received
- concessions made
- concepts adopted from others
- unresolved uncertainties

### Relationship to Perspective

The initial Perspective is historical and immutable.

Agent State is live and mutable.

```text
Perspective Pack
        │
        ▼
Initial Agent State
        │
        ▼
Updated Agent State
        │
        ▼
Final Agent State
```

---

## Contribution

A **Contribution** is a single observable act within a discussion.

It is what an agent says or does in response to the current discussion state.

A Contribution should not merely repeat the agent's initial Perspective. It should respond to the evolving state of the discussion.

### Contribution Types

A Contribution may:

- advance an argument
- challenge a claim
- clarify a distinction
- ask a question
- answer a question
- concede a point
- revise a position
- integrate multiple claims
- expose an assumption
- propose a synthesis

### Responsibilities

A Contribution:

- records the speaker
- records the content
- records the current discussion context
- may trigger state updates
- becomes part of the discussion trace

---

## Discussion

A **Discussion** is an ordered sequence of governed Contributions operating over shared state.

It is not merely a transcript.

A transcript records what was said. A Discussion also includes the state changes produced by what was said.

### Responsibilities

A Discussion:

- begins from a Perspective Pack
- manages Agent States
- manages shared discussion state
- selects or permits speakers
- records Contributions
- updates understanding
- terminates under defined conditions

---

## Discussion State

**Discussion State** is the mutable state of the whole discussion.

It represents what the runtime currently knows about the collective inquiry.

### Responsibilities

Discussion State may include:

- original question
- current round
- transcript
- active agents
- current speaker
- shared claims
- disputed claims
- open questions
- emerging themes
- candidate positions
- areas of convergence
- areas of unresolved disagreement
- current collective understanding

### Distinction from Agent State

Agent State belongs to a participant.

Discussion State belongs to the discussion as a whole.

---

## Shared Claim

A **Shared Claim** is a proposition that has entered the discussion and is available to the group.

It does not have to be accepted by all agents.

### Responsibilities

A Shared Claim:

- captures an assertion made during discussion
- may have supporters and objectors
- may be refined over time
- may contribute to synthesis
- may remain unresolved

---

## Disagreement

A **Disagreement** is a recognised conflict between claims, criteria, interpretations, or priorities.

Disagreement is not a failure of the runtime. It is one of the primary sources of emergence.

### Responsibilities

A Disagreement:

- identifies the point of conflict
- records participating agents or positions
- distinguishes substantive conflict from misunderstanding
- may prompt clarification or synthesis
- may remain unresolved in the final report

---

## Collective Understanding

**Collective Understanding** is the evolving body of meaning produced by the group.

It is not necessarily consensus.

It may include agreement, structured disagreement, clarified criteria, better questions, and recognition of uncertainty.

### Responsibilities

Collective Understanding may contain:

- accepted claims
- contested claims
- refined concepts
- criteria for judgement
- explanatory frames
- unresolved questions
- implications
- minority reports

### Important Distinction

```text
Consensus = agreement
Collective Understanding = improved shared grasp of the issue
```

The runtime should not force consensus unless explicitly configured to do so.

---

## Synthesis

A **Synthesis** is a snapshot of Collective Understanding at a particular point in time.

It is not necessarily a final answer.

It may be provisional, plural, conditional, or explicitly incomplete.

### Responsibilities

A Synthesis should identify:

- what has become clearer
- where agents converged
- where disagreement remains
- which criteria emerged
- which positions strengthened or weakened
- what questions remain open
- what the group would now say in response to the original Question

---

## Emergence

**Emergence** occurs when the discussion produces understanding that was not present in any single initial Perspective.

Emergence may appear as:

- a new distinction
- a reframed question
- an integrated position
- a recognised pattern across perspectives
- a more precise disagreement
- a new evaluative criterion
- a better account of uncertainty

### Non-Examples

The following are not sufficient evidence of emergence:

- a longer transcript
- repeated agreement
- a majority vote
- a summary of initial positions
- a forced compromise

---

## Runtime Boundary

The runtime governs the conditions under which interaction occurs.

It should not prescribe the content of reasoning.

The runtime is responsible for:

- loading a Perspective Pack
- initialising Agent States
- maintaining Discussion State
- selecting or permitting Contributions
- applying governance rules
- recording state transitions
- producing Synthesis artefacts

The runtime is not responsible for:

- deciding the answer in advance
- modifying the original Perspective Pack
- forcing consensus
- hiding disagreement for neatness
- treating transcript length as evidence of progress

---

## Minimal v0.2 Objects

The first implementation should require only these objects:

```text
Question
Perspective
PerspectivePack
AgentState
Contribution
DiscussionState
Synthesis
```

Anything beyond these should be added only when its responsibility is clear.

---

## Design Test

A proposed feature belongs in v0.2 only if it helps answer at least one of these questions:

1. What was the initial state of understanding?
2. What contribution occurred?
3. How did that contribution change agent or group state?
4. What understanding emerged that was not present initially?
5. What remains unresolved?

If a feature cannot answer one of these questions, it should be deferred.

---

## Summary

Agentic Co-Emergence v0.2 models discussion as governed transformation.

```text
Perspective Pack
        │
        ▼
Initial Agent States
        │
        ▼
Governed Contributions
        │
        ▼
State Updates
        │
        ▼
Collective Understanding
        │
        ▼
Synthesis
```

The project should therefore be built around state, transformation, and evidence of emergence — not around producing plausible dialogue for its own sake.
